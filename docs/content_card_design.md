# Модель карточек контента

Этот документ описывает карточную модель Pachuca Archive: какие CSV участвуют, как они связаны и как эти данные отображаются на сайте.

Общий контекст проекта находится в [`project_context.md`](project_context.md).

## Основная идея

Пользователь ищет не конкретную строку из CSV, а понятную сущность: игру, фильм, сериал, аниме, баттл или подборку.

Поэтому сайт использует два уровня:

* материал — конкретная публикация на площадке;
* карточка — пользовательская сущность, объединяющая один или несколько материалов.

Пример:

```text
Карточка:
Resident Evil 7

Материалы:
полный стрим на Boosty
перезалив в Telegram
нарезка на YouTube
```

## CSV-файлы

### `content_items.csv`

Путь:

```text
public/data/content_items.csv
```

Одна строка = один материал.

Материалом может быть:

* видео YouTube;
* пост Boosty;
* перезалив Telegram;
* запись стрима;
* нарезка;
* реакция;
* прохождение;
* разовый заказ или другой самостоятельный фрагмент.

Ключевые поля:

* `content_id` — уникальный ID материала;
* `canonical_title` — нормальное название материала;
* `content_type` — тип контента;
* `content_tags` — теги через запятую;
* `rating` — оценка, если есть;
* `rating_source` — источник оценки;
* `first_date` — первая дата материала;
* `last_date` — последняя дата материала;
* `duration_hours` — длительность в часах;
* `youtube_url` — ссылка на YouTube;
* `telegram_url` — ссылка на Telegram;
* `boosty_url` — ссылка на Boosty.

### `content_cards.csv`

Путь:

```text
public/data/content_cards.csv
```

Одна строка = одна карточка контента.

Ключевые поля:

* `card_id` — уникальный ID карточки;
* `card_title` — название карточки для интерфейса;
* `content_type` — основной тип карточки;
* `content_tags` — теги карточки;
* `rating` — итоговая оценка карточки;
* `rating_source` — источник итоговой оценки;
* `is_active` — `1` или `0`, позволяет скрыть карточку без удаления.

### `content_card_items.csv`

Путь:

```text
public/data/content_card_items.csv
```

Таблица связей между карточками и материалами.

Ключевые поля:

* `card_id` — ID карточки из `content_cards.csv`;
* `content_id` — ID материала из `content_items.csv`;
* `sort_order` — порядок материала внутри карточки;
* `card_name` — служебное поле для ручной работы, не является источником истины.

## Важные правила модели

`content_items.csv` остаётся первичным источником материалов. Карточки не заменяют материалы, а группируют их для интерфейса.

Карточка может содержать один материал. В этом случае карточка на сайте сразу открывает источник.

Карточка может содержать несколько материалов. В этом случае карточка раскрывается и показывает список источников.

Один материал может входить в несколько карточек. Это нужно для мультиконтентных стримов, где в одной записи были, например, фильм и игра.

Не каждый материал обязан иметь ручную карточку. Если материал не связан с `content_cards.csv`, код сайта создаёт для него одиночную карточку автоматически.

Карточки с `is_active = 0` не показываются.

## Как сайт собирает карточки

Логика находится в:

```text
src/lib/contentCards.ts
```

Сборка происходит так:

1. Читаются `content_items.csv`, `content_cards.csv` и `content_card_items.csv`.
2. Материалы индексируются по `content_id`.
3. Активные карточки связываются с материалами через `content_card_items.csv`.
4. Материалы внутри карточки сортируются по `sort_order`, затем по дате.
5. Материалы без ручной карточки превращаются в одиночные карточки.
6. Итоговый список сортируется по самой поздней дате материала внутри карточки.

### Авторский контент

Раздел `Авторский контент` собирается отдельно от основного архива и не выводится автоматически из `content_tags` обычных карточек.

Для него используются ручные таблицы:

```text
public/data/creator_cards.csv
public/data/creator_card_items.csv
```

`creator_cards.csv` описывает карточки авторского раздела. `creator_card_items.csv` связывает эти карточки с существующими `content_id` из `content_items.csv`. Поля `creator_item_title` и `creator_item_date` в связях служебные и сайтом не используются.

`creator_tags` задаёт разделы внутри авторского контента и используется для фильтра. Сортировка карточек остаётся общей: по самой поздней дате материала внутри карточки.

## Отображение на сайте

Основной компонент:

```text
src/components/cardsView.tsx
```

Сетка:

* desktop — две карточки в ряд;
* mobile — одна карточка в ряд.

На карточке показываются:

* тип контента;
* дата последнего обновления;
* название;
* оценка, если есть;
* теги;
* иконки доступных площадок.

## Сценарии карточек

### Один материал

Если после фильтрации в карточке остался один материал, клик по карточке открывает источник в новой вкладке.

### Несколько материалов

Если материалов несколько, карточка раскрывается и показывает список материалов.

### Несколько источников

Если внутри карточки есть несколько площадок и на одной или нескольких площадках больше одного материала, материалы группируются по источникам.

При открытии карточки или группы источника сайт сохраняет позицию раскрываемого заголовка на экране. Перед изменением состояния UI запоминает положение заголовка, после рендера компенсирует изменение высоты страницы через `window.scrollBy`. Это нужно для длинных списков материалов: если один большой список закрывается, а другой открывается, пользователь остаётся рядом с местом клика, без принудительного скролла к началу блока.

## Фильтрация

Фильтр по площадке работает по материалам внутри карточки. Если для выбранной площадки у карточки нет материалов, карточка скрывается.

Фильтр по типу работает по `content_type` карточки.

Вкладка `Авторский контент` работает только по ручным карточкам из `creator_cards.csv`. Обычные `content_cards.csv` в эту вкладку не подмешиваются. Внутренний фильтр вкладки строится по `creator_tags`.

Для баттлрэпа есть отдельные фасетные фильтры по баттл-площадке и, где размечено, по ивенту. Они работают поверх `content_tags` карточки и вложенных материалов.

Сейчас вложенные ивент-фильтры описаны для `Кубок МЦ` (`kbk_*`) и `RBL` (`rbl_*`). Остальные площадки фильтруются на уровне площадки, пока их ивенты не размечены последовательно.

Если баттлрэп-фильтр оставляет внутри карточки-контейнера ровно один видимый материал, UI показывает этот материал как результат фильтрации:

* название берётся из `item.title`;
* дата берётся из `item.lastDate || item.firstDate`;
* теги берутся из `item.tags`.

Это не меняет карточную модель и CSV-связи. Карточка остаётся контейнером, но в фильтрованной выдаче одиночное совпадение отображается как конкретный найденный материал, чтобы общий заголовок контейнера не вводил в заблуждение.

## Поиск

Основная логика поиска находится в:

```text
src/components/cardsView.tsx
```

Поиск работает поверх уже собранных карточек и учитывает выбранную площадку. Если выбран фильтр `YouTube`, `Boosty` или `Telegram`, поиск проверяет только материалы этой площадки.

### Уровни совпадений

Результаты делятся на `strong` и `weak`.

`strong` означает, что запрос совпал с самой карточкой. Такие карточки считаются основными результатами.

Поля `strong`-поиска:

* название карточки `card.title`;
* технический тип карточки `card.contentType`;
* человекочитаемый label типа из `src/lib/contentMeta.ts`;
* raw-теги карточки из `card.tags`;
* человекочитаемые labels тегов из `src/lib/tagLabels.ts`.

`weak` означает, что запрос не совпал с карточкой напрямую, но совпал с одним или несколькими вложенными материалами.

Поля `weak`-поиска:

* название материала `item.title`;
* технический тип материала `item.contentType`;
* человекочитаемый label типа;
* raw-теги материала;
* человекочитаемые labels тегов материала.

Эти поля не участвуют в общем текстовом поиске:

* `rating`;
* `firstDate`;
* `lastDate`;
* `durationHours`;
* `youtubeUrl`;
* `telegramUrl`;
* `boostyUrl`.

Они исключены намеренно: рейтинг, даты и длительность создают шумные числовые совпадения, а технические ID в URL создавали ложные совпадения вроде `fb4` внутри UUID Boosty.

### Отображение weak-результатов

Если карточка найдена как `weak`, в раскрытии показываются только совпавшие вложенные материалы.

Например, если большая ретроспективная карточка найдена только потому, что один материал внутри неё содержит запрос, UI показывает этот один материал, а не всю ретроспективу.

Для `weak`-карточек дополнительно показывается подпись:

```text
Найдено внутри: X из Y
```

`weak`-карточка с одним совпавшим материалом раскрывается как карточка, а не открывает ссылку сразу. Это нужно, чтобы пользователь видел контекст совпадения.

### Удаление дублей между strong и weak

`weak`-результаты не скрываются глобально при наличии `strong`-результатов.

Вместо этого применяется более точное правило:

1. Сначала собираются все `content_id`, которые уже входят в `strong`-результаты.
2. Из `weak`-карточек удаляются совпавшие материалы с этими `content_id`.
3. Если после удаления дублей в `weak`-карточке не осталось материалов, карточка скрывается.

Это сохраняет полезные weak-результаты для широких запросов по персонам или темам, но убирает дубли вроде одной и той же записи в отдельной карточке и в ретроспективе.

### Правила текстового совпадения

Поиск не чувствителен к регистру.

Сначала проверяется прямое вхождение всего запроса в поле.

Если прямого вхождения нет, запрос разбивается на части по пробелам и пунктуации. Поле тоже разбивается на части. Совпадение засчитывается, если каждая часть запроса совпадает с началом одной из частей поля.

Пример:

```text
Запрос: 17 не
Поле: 17й Независимый (17IB)
Результат: совпадает
```

При этом запрос не должен цеплять середину слов или годов:

```text
Запрос: 17 не
Поле: Валериан и город тысячи планет (2017)
Результат: не совпадает
```

### Технические поля

Для raw-тегов и raw-типов используется более строгий матчинг, чем для человекочитаемых названий.

Технические поля не проверяются свободным `includes`. Каждая часть запроса должна совпасть с началом части технического значения.

Примеры:

```text
10 -> zom100: не совпадает
100 -> zom100: не совпадает
fb4 -> fb4: совпадает
140 -> 140: совпадает
battle rap -> battle_rap: совпадает
```

Это нужно, чтобы короткие числовые запросы не находили технические теги случайно.

### Рейтинг, даты и длительность

Рейтинг, даты и длительность не участвуют в общем текстовом поиске.

UX-решение на будущее: если понадобится навигация по этим полям, лучше вынести их в отдельные фильтры или сортировки, потому что числовые запросы чаще означают год, выпуск, сезон, эпизод или часть названия.

## Сортировка

Поддерживаются варианты:

* новые сверху;
* старые сверху;
* оценка по убыванию;
* оценка по возрастанию;
* название от А до Я;
* название от Я до А.

Для сортировки по дате используется максимальная дата среди материалов карточки:

```text
MAX(last_date или first_date)
```

Сортировка по оценке работает только для рейтингуемых типов `movie`, `series` и `anime`.

Для сортировки используется одна однозначная числовая оценка:

1. `content_cards.csv.rating`, если он заполнен числом.
2. Единая или единственная числовая оценка видимых материалов, если рейтинга карточки нет.

Значение `-`, разные оценки внутри карточки и карточки нерейтингуемых типов не участвуют в числовой сортировке и уходят ниже. При одинаковой оценке используется сортировка по дате: новые сверху.

## Рейтинг

Рейтинг может храниться на двух уровнях.

В UI рейтинги отображаются только для рейтингуемых типов контента:

* `movie`;
* `series`;
* `anime`.

Если оценка технически заполнена у материала или карточки другого типа, она не показывается в интерфейсе. Это защищает смешанные карточки и разделы вроде игр, баттлрэпа, ИРЛ, видео и дегрода от случайного отображения оценок.

`content_cards.csv.rating` — оценка всей карточки, если она задана вручную на уровне карточки.

`content_items.csv.rating` — оценка конкретного материала, сезона, части или записи.

Значение `-` в поле `rating` означает осознанное отсутствие оценки. В UI оно отображается как:

```text
Без оценки
```

Если у карточки есть `content_cards.csv.rating`, он имеет приоритет над оценками материалов. UI показывает на карточке:

```text
Оценка: X/10
```

Если у карточки есть `content_cards.csv.rating`, оценки рядом с материалами скрываются, чтобы не дублировать оценку карточки.

Если `content_cards.csv.rating` пустой, но среди видимых материалов есть ровно одна оценка, UI поднимает её на карточку:

```text
Оценка: X/10
```

Если `content_cards.csv.rating` пустой, но у всех видимых материалов карточки есть одинаковая оценка, UI тоже показывает эту оценку на карточке:

```text
Оценка: X/10
```

В этом случае оценки рядом с материалами скрываются, чтобы не дублировать одно и то же значение.

Если `content_cards.csv.rating` пустой, а внутри карточки есть несколько разных оценок или оценена только часть материалов несколькими оценками, на карточке показывается счётчик:

```text
Оценки: N
```

А конкретные оценки показываются в раскрытом списке материалов рядом с соответствующими источниками.

Если материал не имеет ручной карточки и превращается в одиночную карточку автоматически, рейтинг берётся из самого материала.

Это нужно для сериалов и аниме, где оценки могут быть у отдельных сезонов или частей, но итоговой оценки всей карточки ещё нет.

## Что пока не является частью модели

Сейчас не используются:

* `relation_type`;
* `display_title` для связи;
* `is_primary`;
* отдельные страницы карточек как основной сценарий;
* обложки карточек;
* рекомендации похожего контента;
* админка для модерации карточек.

Эти поля и сценарии можно добавить позже, когда появится реальная необходимость.

## Риски и проверки данных

Для стабильной работы сайта желательно автоматически проверять:

* ссылки из `content_card_items.csv` на отсутствующие `content_id`;
* ссылки на отсутствующие `card_id`;
* дубли тегов внутри одной карточки;
* неизвестные значения `content_type`;
* пустые `card_title` и `canonical_title`;
* невалидные даты;
* материалы без ссылок на площадки.
