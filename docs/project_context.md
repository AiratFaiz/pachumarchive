# Контекст проекта

Этот документ описывает текущее состояние Pachuca Archive: цель проекта, источники данных, пайплайн обновления и устройство сайта.

Подробная спецификация карточной модели находится в [`content_card_design.md`](content_card_design.md).

## Цель

Pachuca Archive — архив просмотренного и пройденного контента Пачуки.

Основные пользовательские сценарии:

* найти конкретную игру, фильм, сериал, аниме, баттл или другой контент;
* понять, смотрелся ли этот контент на стримах;
* открыть оригинальный материал на YouTube, Telegram или Boosty;
* увидеть несколько материалов вокруг одной сущности в одной карточке.

## Текущий стек

* Next.js
* React
* TypeScript
* Tailwind CSS
* Vercel
* CSV-файлы в репозитории как источник данных сайта

## Где что лежит

```text
src/app/                 страницы Next.js App Router
src/components/          клиентские и серверные UI-компоненты
src/lib/                 чтение CSV и подготовка данных
public/data/             CSV, которые попадают в деплой и читаются сайтом
scripts/                 скрипты сбора, ревью и импорта данных
data/                    рабочие черновики и файлы модерации
docs/                    документация проекта
```

Служебные файлы для AI-агентов:

```text
AGENTS.md
CLAUDE.md
```

Они не являются пользовательской документацией проекта.

## Модель данных

Сайт сейчас построен вокруг карточек контента.

Основные таблицы:

```text
public/data/content_items.csv
public/data/content_cards.csv
public/data/content_card_items.csv
```

`content_items.csv` — первичная таблица материалов. Одна строка описывает конкретную публикацию или запись с площадки.

Примеры материалов:

* видео YouTube;
* пост Boosty;
* перезалив Telegram;
* отдельная запись стрима;
* нарезка;
* реакция;
* прохождение.

`content_cards.csv` — таблица карточек. Одна строка описывает пользовательскую сущность: игру, фильм, сериал, аниме, баттл, подборку или другой самостоятельный контент.

`content_card_items.csv` — таблица связей между карточками и материалами. Один материал может входить в несколько карточек, если стрим был мультиконтентным.

Идентификация:

```text
content_id
canonical_title
```

Категории:

```text

content_tags
```

Даты:

```text

last_date
```

Метаданные:

```text
duration_sec
duration_hours
records_count
sources
rating
rating_source

```

Ссылки:

```text

```

Сырые описания источников:

```text

```

## Типы контента

Основные значения `content_type`:

* `game`
* `movie`
* `series`
* `anime`
* `battle_rap`
* `music`
* `degrod`
* `irl`
* `videos`
* `unknown`

Теги хранятся в `content_tags` как список через запятую. Человекочитаемые названия тегов сейчас находятся в `src/lib/tagLabels.ts`.

## Источники данных

### YouTube

Каналы:

* `PAPAPACHUCA`
* `GUZ3LKA`

Собираются видео и стримы. Для части контента используется `yt-dlp` и cookies для доступа к материалам с ограничениями.

### Boosty

Источник:

```text
guzelka_games
```

Сбор идёт через API Boosty. Локальные токены должны храниться в `.env` и не коммититься.

### Telegram

Источник:

```text
gzlkgms
```

Пока используется ручной экспорт из Telegram Desktop в `data/telegram/result.json`.

## Пайплайн обновления данных

Общая схема:

```text
YouTube / Boosty / Telegram
        ↓
data/content_drafts.csv
        ↓
ручная модерация
        ↓
public/data/content_items.csv
        ↓
content_cards.csv + content_card_items.csv
        ↓
сайт
```

Новые записи сначала попадают в `data/content_drafts.csv`. Перед импортом вручную проверяются предложенные поля:

```text
suggested_canonical_title
suggested_content_type
suggested_content_tags
```

После проверки строка получает статус:

```text
approved
```

Затем импортируется в `public/data/content_items.csv`.

## Рабочие скрипты

Сбор новых записей YouTube и Boosty:

```bash
python scripts/collect_new_content.py
```

Подготовка Telegram-черновиков из актуального экспорта:

```bash
python scripts/telegram_drafts.py
```

Ревью черновиков:

```bash
python scripts/review_drafts.py
```

Импорт утверждённых черновиков:

```bash
python scripts/apply_drafts.py
```

Генерация или обновление карточек для части контента:

```bash
python scripts/generate_movie_cards.py
```

Проверка структуры CSV-данных:

```bash
npm run validate:data
```

## Как сайт читает данные

Текущая главная страница:

```text
src/app/page.tsx
```

Основной серверный загрузчик данных:

```text
src/lib/contentCards.ts
```

Справочники интерфейса для типов контента, вкладок, сортировки и площадок:

```text
src/lib/contentMeta.ts
```

Справочник человекочитаемых названий тегов хранится отдельно:

```text
src/lib/tagLabels.ts
```

Типы контента и теги могут пересекаться по ключам, но используются в разных UI-контекстах, поэтому эти справочники намеренно не объединены.

Основной клиентский компонент отображения:

```text
src/components/cardsView.tsx
```

Вложенные UI-компоненты карточек:

```text
src/components/cards/
```

Вспомогательная логика отображения карточек:

```text
src/lib/contentCardView.ts
```

Поток данных:

```text
CSV в public/data
        ↓
getContentCards()
        ↓
HomePage
        ↓
CardsView
```

## Что реализовано на сайте

* карточки контента;
* поиск по названию карточки и материалам внутри неё;
* фильтр по типу контента;
* фильтр по площадке;
* сортировка по дате и названию;
* раскрытие карточки со списком материалов;
* группировка материалов по источникам для больших карточек;
* переход к оригинальным материалам.

## Текущие ограничения

* часть данных ещё не полностью очищена;
* возможны дубли карточек, тегов или материалов;
* не все оценки заполнены;
* не все материалы объединены в карточки вручную;
* отдельная страница контента пока не реализована;
* в CSV местами встречаются технические проблемы вроде дублирующихся заголовков.

## Ближайшие структурные задачи

* продолжить дробление `src/components/cardsView.tsx`, если появятся новые независимые UI-блоки;
* расширить проверку CSV на битые связи, неизвестные теги, дубли заголовков и пустые поля;
* если понадобится детальная страница контента, добавить новый маршрут уже поверх реальной карточной модели.
